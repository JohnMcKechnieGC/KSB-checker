# Theme 1: Lifecycle and Methodology

## K1: All stages of the Software Development Life Cycle (inputs/outputs)

Below is how we apply SDLC end-to-end for *CareFlow Portal*.

### 1) Discovery / Initiation

* **Purpose:** Confirm problem, value, feasibility, and constraints.
* **Inputs:** Business goals, current process pain points, initial stakeholder interviews, high-level risks (GDPR, clinical safety).
* **Activities:** Workshops, problem statements, success metrics, feasibility spikes.
* **Outputs:** Project brief, high-level scope, risk register draft, success measures (e.g., reduce appointment no-shows by X%).

### 2) Requirements / Analysis

* **Inputs:** Stakeholder needs, regulatory constraints, existing system integrations, user feedback.
* **Activities:** User stories, use cases, acceptance criteria, data mapping, non-functional requirements (performance, availability, security).
* **Outputs:** Prioritised backlog, refined scope, NFR list, definition of done, traceability notes (story ↔ requirement).

### 3) Design (Solution + UX/UI)

* **Inputs:** Requirements, constraints (security, maintainability), existing architecture standards, design system.
* **Activities:** Wireframes, component design, API contracts, ERD updates, threat modelling (lightweight), sequence diagrams.
* **Outputs:** UX prototypes, architecture diagram, API specs (OpenAPI), database design updates, test approach outline.

### 4) Build / Implementation

* **Inputs:** Approved designs, backlog items, coding standards, branching strategy, CI pipeline.
* **Activities:** Code changes, peer reviews, unit tests, feature toggles, documentation updates.
* **Outputs:** Working increments in a feature branch, reviewed PRs, updated docs, build artifacts (container/image/package).

### 5) Verification / Testing

* **Inputs:** Built increments, acceptance criteria, test plan, test data, environments.
* **Activities:** Unit/integration/system tests, security scanning, performance checks, defect triage.
* **Outputs:** Test evidence, defect list, release candidate sign-off recommendation.

### 6) Release / Deployment

* **Inputs:** Approved release candidate, change notes, deployment checklist, rollback plan.
* **Activities:** CI/CD to staging then production, smoke tests, monitoring setup.
* **Outputs:** Production release, release notes, updated runbooks, monitoring dashboards.

### 7) Operations / Maintenance

* **Inputs:** Logs/metrics, support tickets, user feedback, incident reports.
* **Activities:** Bug fixes, minor enhancements, dependency patching, performance tuning.
* **Outputs:** Patches, updated backlog, incident post-mortems, knowledge base updates.

### 8) Retirement / Decommission

* **Inputs:** Product strategy, contract end, replacement system readiness, data retention rules.
* **Activities:** Migration planning, archival, user comms, access removal.
* **Outputs:** Archived data (policy-compliant), system shutdown, final report.

✅ **Pass coverage (K1):** All SDLC stages described with what they contain and their inputs/outputs.

## K5: Similarities & differences between methodologies (Agile vs Waterfall)

### Similarities

* Both aim to deliver working software that meets user needs.
* Both rely on requirements, design, build, test, and deployment (just structured differently).
* Both need governance: risk management, stakeholder sign-off points, documentation (amount varies).

### Key differences

**Waterfall**

* **Flow:** Sequential phases; changes later are costly.
* **Strengths:** Predictable milestones, clear documentation/sign-offs; useful for fixed-scope/regulatory-heavy work.
* **Risks:** Late feedback; may deliver the “wrong thing” if requirements change or were misunderstood early.

**Agile (Scrum/Kanban)**

* **Flow:** Iterative increments; frequent feedback and reprioritisation.
* **Strengths:** Early value delivery, adaptable to change, continuous stakeholder engagement.
* **Risks:** Needs strong product ownership; scope can drift if priorities/definition of done aren’t managed.

### How we apply in practice (hybrid)

* We use **Agile Scrum** for delivery (2-week sprints, demos, backlog refinement).
* We keep **Waterfall-like governance** for compliance items: security review gates, release approvals, documented DPIA-style checks for sensitive changes.

✅ **Pass coverage (K5):** Similarities and differences described with examples.

# Theme 3: Reuse

## K7: Design approaches & patterns for reusable solutions (bespoke and off-the-shelf)

### Design approaches used

* **Layered architecture:** UI → service → data access to keep concerns separate and reusable.
* **Component-based UI:** reusable appointment card, patient banner, and validation components.
* **API-first contracts:** stable endpoints allow reuse across web app and internal tools.

### Patterns applied (examples)

* **Repository pattern:** consistent data access for appointments/patients.
* **Strategy pattern:** selectable reminder channels (SMS/email) with a shared interface.
* **Factory pattern:** building notification payloads based on clinic configuration.
* **Observer/Event-driven style:** emitting events like `AppointmentUpdated` for downstream processing.

### Reuse options evaluated

* **Off-the-shelf:** notification provider SDK, auth library (OIDC), validation library
* **Bespoke:** internal shared “audit logging” module to meet compliance requirements

✅ **Pass coverage (K7):** Suggests and applies design approaches/patterns and includes bespoke/off-the-shelf.

## Distinction: Evaluate and recommend reusable approaches (K7)

### Example decision: “Build vs Buy” for audit logging

* **Option A (off-the-shelf):** faster initial setup but limited audit schema control, uncertain alignment with NHS reporting expectations, potential vendor lock-in.
* **Option B (bespoke shared module):** more effort upfront, but consistent audit format, tailored retention policies, easier internal reuse across products.

**Recommendation:** Bespoke shared module for audit logging (compliance-critical) + off-the-shelf for commodity concerns (auth, UI components, standard SDKs).
**Justification criteria used:** security/compliance fit, long-term maintainability, licensing risk, cost, and integration effort.

✅ **Distinction coverage (K7):** Evaluates options and gives a justified recommendation.

## K10: Relational and non-relational database principles and uses

### Relational (PostgreSQL)

* Used for appointments/patients due to:

  * **ACID transactions** (avoid double booking)
  * **Structured schema** and constraints (foreign keys, uniqueness)
  * **Normalization** for consistent data and reporting
  * Indexing for search performance

### Non-relational (MongoDB / Redis-style use cases)

* Considered/used for:

  * **Document store:** flexible schemas for audit/event payloads (if requirements change often)
  * **Cache (Redis):** speeding up reference lookups and reducing load on primary DB
* Trade-offs understood:

  * Eventual consistency and schema flexibility vs strict constraints and joins
